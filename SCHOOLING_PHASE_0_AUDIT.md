## What exists today

### Routes
- `/schooling` — board catalogue (21 boards in 4 types)
- `/schooling/[slug]` — per-board page with official site link, syllabus
  link (where available), sample-paper link (where available)

**That's it.** No per-class, no per-subject, no per-chapter routes.

### Data
- `src/lib/schooling-data.ts` — 21 boards as TypeScript constants. Each
  board has: `slug`, `name`, `state`, `classes[]`, `language[]`,
  `websiteUrl`, `syllabusUrl?`, `samplePaperUrl?`, `blurb`.
- `CLASS_11_12_STREAMS` constant — science/commerce/humanities subject
  lists, but these are just display strings, not entry points to data.
- **No** subject / chapter / quiz data of any kind.

### Database tables relevant to schooling
- **None named "schooling".** BUT — and this is the headline finding —
  the existing `Subject` + `Topic` schema (tied to `Exam`) already has
  exactly the shape the brief asks for chapter pages:

```
Topic {
  id, subjectId, parentId, code, name, description,
  syllabusText,    // detailed syllabus content for AI context
  notes,           // markdown: Overview, Key Concepts, Formulas,
                   //          Worked Examples, Common Mistakes, Quick Reference
  notesGeneratedAt, notesGeneratedBy, weight, orderIdx,
  parent, children, questions, weaknessMaps
}
```

The `Topic.notes` field already stores the **6-section markdown
structure** your brief asks for per chapter (Overview / Key Concepts /
Formulas / Worked Examples / Common Mistakes / Quick Reference) —
generated by `scripts/generate-topic-notes.ts` which is live and works.

### Existing pipelines that already work for schooling content
- **Note generation** (`scripts/generate-topic-notes.ts`) — Claude-based,
  prompt structured to the exact format the brief requires. Already
  budget-capped, idempotent, resumable.
- **Practice question generation** (`scripts/generate-questions.ts`)
  — the PYQ generator. Generates exam-style questions per topic.
- **Mock player + adaptive engine** (`/mocks/[id]`) — already serves
  multi-question tests against any topic set, scores them, shows weak
  areas.
- **WeaknessMap** — already tracks per-user weak topics across subjects.
- **QuestionTranslation cache** — already translates questions on demand
  into 18 Indian languages.
- **Verification badges** — already work on any Fact row.

### Existing pipelines that DO NOT exist for schooling
- Parent account type / parent dashboard / child-parent linking
- Progress tracking dashboard (the existing `WeaknessMap` is exam-
  oriented, not study-trajectory-oriented across chapters)
- "Streak / days studied" tracking
- Class × Board mapping to subjects (because the existing schema
  attaches subjects to entrance Exams, not to school boards/classes)

## The big question this audit raises

**Reuse the Exam/Subject/Topic schema for school content, or build
parallel SchoolClass/SchoolSubject/SchoolChapter tables?**

### Option A — Reuse (recommended)

Treat each `(Board, Class)` pair as a new `Exam` row. e.g.:
- `Exam(code="SCHOOL_CBSE_10", category=SCHOOL_BOARD, shortName="CBSE Class 10")`
- `Subject(examId=SCHOOL_CBSE_10, code="MATH", name="Mathematics")`
- `Topic(subjectId=…, code="ch1_real_numbers", name="Real Numbers")` →
  this is a Chapter
- Existing `Topic.notes` field stores the chapter's concept summary
- Existing `Question` + `Mock` + `Attempt` engine generates and runs
  the mastery quiz

Add `SCHOOL_BOARD` (or `SCHOOLING`) to the `ExamCategory` enum.

**Pros:**
- Zero new tables. Schema migration is 1 enum value addition.
- `generate-topic-notes.ts` works as-is. Generates the chapter summary
  in the exact 6-section format your brief asks for.
- `generate-questions.ts` works as-is. Generates practice questions and
  mastery-quiz items.
- Mock player works as-is. The "10-question mastery quiz" UI already
  exists at `/mocks/[id]`.
- WeaknessMap works as-is. "Weak chapters" surfaces naturally per user.
- Translation cache works as-is. The 1500-Hindi-rows we already have
  apply only to entrance-exam questions; new school questions get
  translated through the same mechanism.
- Verification badges work as-is. Each chapter's notes get a
  `Fact` row pointing at the NCERT source.

**Cons:**
- Conceptual overlap: an Exam row that isn't really an exam. Could
  confuse the existing /exams browse pages. (Fix: filter out
  `category=SCHOOL_BOARD` from the `/exams` UI; surface only on
  /schooling URLs.)
- The "Class" concept doesn't fit cleanly into the Exam model — we'd
  encode it in `Exam.shortName` and `Exam.code` (e.g., "CBSE_10").

### Option B — Parallel schema

New tables: `SchoolClass`, `SchoolSubject`, `SchoolChapter`,
`SchoolQuiz`, `SchoolProgress`, `ParentLink`.

**Pros:** Cleaner mental model. No conceptual overlap.
**Cons:** Duplicates everything we just built. ~10× the work. Note
generation script needs a fork. Mock player needs a fork or a
generic abstraction. WeaknessMap concept needs duplicating. Months,
not weeks.

### My recommendation

**Option A**, gated on you confirming that's acceptable.

The conceptual overlap is a small cost. The infrastructure win is
massive — we ship the spec in days instead of months.
